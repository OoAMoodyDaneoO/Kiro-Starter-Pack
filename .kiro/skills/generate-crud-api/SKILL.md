## Description

Generates a complete set of CRUD (Create, Read, Update, Delete) API endpoints for a given resource. The output includes a data model, a controller with business logic, route definitions, request validation, and consistent error handling. All generated code follows project conventions and is ready to integrate into an existing service.

When invoked, you'll need: the resource name in singular form (e.g., `product`, `user`, `order`), field definitions in `name:type` format (e.g., `title:string, price:number, active:boolean`), and the database backend (e.g., `postgres`, `mongodb`, `dynamodb`, `mysql`). Optionally specify the programming language, web framework, and URL base path (default: `/api`).

## Steps

1. **Generate data model** — Create a model file for the resource in the project's models directory:
   - Define the schema with all specified fields, types, and constraints
   - Add an auto-generated primary key field (`id`) and timestamp fields (`createdAt`, `updatedAt`)
   - Include database-specific annotations or decorators for the chosen database type
   - Add a module-level comment describing the model and its fields

2. **Generate controller** — Create a controller file with handler functions for each CRUD operation:
   - `create` — accept validated input, persist to database, return the created resource with `201` status
   - `getById` — look up a single resource by ID, return `404` if not found
   - `list` — return a paginated list of resources with optional query filters
   - `update` — accept partial input, update the existing resource, return `404` if not found
   - `delete` — remove the resource by ID, return `404` if not found, return `204` on success
   - Each handler delegates to a service or repository layer rather than querying the database directly

3. **Generate route definitions** — Create a routes file that maps HTTP methods and paths to controller handlers:
   - `POST   <base_path>/<resource_name>s`       → `create`
   - `GET    <base_path>/<resource_name>s/:id`    → `getById`
   - `GET    <base_path>/<resource_name>s`        → `list`
   - `PUT    <base_path>/<resource_name>s/:id`    → `update`
   - `DELETE <base_path>/<resource_name>s/:id`    → `delete`
   - Register the routes with the application's router or framework-specific routing mechanism

4. **Add request validation** — Create validation schemas for incoming request bodies:
   - `create` schema — require all mandatory fields, enforce type and length constraints
   - `update` schema — allow partial input, validate only the fields that are present
   - Validate path parameters (e.g., `id` format) and query parameters (e.g., pagination limits)
   - Return structured `400` error responses with field-level error details on validation failure

5. **Add error handling** — Wire up consistent error handling across all generated endpoints:
   - Map common failure cases to appropriate HTTP status codes (`400`, `404`, `409`, `500`)
   - Return structured JSON error responses with `error`, `message`, and `details` fields
   - Handle database-specific errors (duplicate key, connection failure) with meaningful messages
   - Log errors with request context (request ID, resource ID, operation) for debugging

## Expected Output

A set of files implementing a fully functional CRUD API for the specified resource: a data model with schema definitions, a controller with five endpoint handlers, a routes file mapping HTTP verbs to handlers, validation schemas for create and update operations, and error handling that returns structured JSON responses. The generated code compiles, integrates with the existing service, and is ready to handle requests after running any necessary database migrations.

## Notes

- The generated code assumes a service or repository pattern is already in place. If not, the controller will include inline data access logic that you should refactor into a separate layer.
- Database migrations are not generated by this skill. Use the `generate-db-migration` skill to create the corresponding table or collection.
- Authentication and authorization middleware are not included. Add route-level guards or middleware after generation to protect endpoints as needed.
- The `list` endpoint includes basic pagination (limit/offset) but not advanced filtering or sorting. Extend the query logic to match your API requirements.
- Field types in the `fields` input are mapped to language-native types. Supported types include `string`, `number`, `boolean`, `date`, and `json`. For custom or complex types, adjust the generated model after creation.
