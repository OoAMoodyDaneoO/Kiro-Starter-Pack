## Description

Generates a database migration file from a human-readable schema change description. The output includes a properly named migration file with `up` and `down` methods that apply and revert the change respectively. The generated SQL is validated for syntax correctness and checked for common pitfalls such as missing indexes, unsafe column renames, or data-loss operations. The migration integrates with the project's existing migration tool and follows its file naming and structure conventions.

When invoked, you'll need: a plain-language description of the schema change (e.g., `add email column to users table`), the target database engine (e.g., `postgres`, `mysql`, `sqlite`, `mssql`), and the migration framework (e.g., `knex`, `prisma`, `sequelize`, `flyway`, `alembic`, `django`, `typeorm`, `drizzle`). Optionally specify a migration name and target schema (default: `public`).

## Steps

1. **Generate migration file** — Create a new migration file in the project's migrations directory following the tool's naming convention:
   - Use the tool's timestamp or version prefix format (e.g., `20240115120000_add_email_to_users.sql` for Flyway, `migrations/001_add_email_to_users.ts` for Knex)
   - Place the file in the correct directory for the chosen migration tool (e.g., `migrations/`, `prisma/migrations/`, `alembic/versions/`)
   - Add a header comment with the migration description, author placeholder, and creation timestamp

2. **Add up method** — Write the forward migration logic that applies the schema change:
   - Translate the schema change description into the appropriate DDL statements for the target database type
   - Use the migration tool's API or raw SQL syntax as appropriate (e.g., `knex.schema.alterTable()` for Knex, raw SQL for Flyway)
   - Include sensible defaults for new columns (e.g., `NOT NULL` with a default value, or `NULL` if no default is appropriate)
   - Add indexes for foreign key columns and frequently queried fields
   - Wrap multiple statements in a transaction where the database and tool support it

3. **Add down method** — Write the rollback migration logic that reverts the schema change:
   - Generate the inverse operation for every statement in the `up` method (e.g., `DROP COLUMN` to undo `ADD COLUMN`, `DROP TABLE` to undo `CREATE TABLE`)
   - Preserve data safety — warn with a comment if the rollback is destructive (e.g., dropping a column loses data)
   - For irreversible changes, add a comment explaining why the down migration cannot fully restore the previous state

4. **Validate SQL** — Check the generated migration for correctness and common issues:
   - Verify SQL syntax is valid for the target database type dialect
   - Check for missing `NOT NULL` constraints on columns that should not be nullable
   - Warn if a column rename or type change could cause data loss
   - Verify that foreign key references point to existing tables and columns (based on available schema context)
   - Flag long-running operations on large tables (e.g., adding a column with a default on Postgres < 11) and suggest safe alternatives

## Expected Output

A timestamped migration file in the project's migrations directory containing `up` and `down` methods (or equivalent forward/rollback sections). The `up` method applies the described schema change with proper types, constraints, and indexes. The `down` method reverts the change safely. The file follows the chosen migration tool's conventions and is ready to run with the tool's migrate command.

## Notes

- The generated migration assumes the target table already exists unless the schema change explicitly describes creating a new table. Verify the current schema before running.
- For tools that use a declarative schema (e.g., Prisma), this skill generates the schema change and the corresponding migration file. Update the schema file first, then run the tool's migration generator to keep them in sync.
- Data migrations (backfilling values, transforming existing rows) are not generated by default. Add a separate step in the `up` method if the schema change requires data transformation.
- Always review the `down` method before relying on it in production. Some changes (dropping a table, removing a column) are inherently destructive and cannot be fully reversed.
- For large tables in production, consider wrapping the migration in a deployment strategy that avoids locking (e.g., `CREATE INDEX CONCURRENTLY` on Postgres, online DDL on MySQL).
- Test migrations against a local or staging database before applying them to production.
